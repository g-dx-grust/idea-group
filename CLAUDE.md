# CLAUDE.md — グラスト 共通開発ルール（GRUST Common Rules）

このファイルは、**株式会社グラスト（GRUST.inc）が開発するすべての業務システム**を
Claude Code で開発する際の**最上位ルールブック**です。
G-DX 配下の全プロダクト（g-dx_kanri / G-HR / Luxates / RGD / dreaMs / G-PO ほか今後新設するシステム）に
横断適用されます。

各システム固有のルール（要件定義の参照先・ロール定義・リリース順など）は、
**各リポジトリの CLAUDE.md** で本ドキュメントを参照したうえで上書き／追記してください。
本ドキュメントと矛盾するシステム固有ルールを書かないでください。

---

## 0. 最重要原則（必ず守る）

1. **"生成AIっぽさ"を出さない**
   いずれのシステムも業務用 SaaS として長期運用される。流行のビジュアル
   （ネオンカラーのグラデーション、過剰な `rounded-2xl`、ガラス風、ネオモーフィズム、
   紫ピンクの配色、過剰なアニメーション、巨大ヒーロー）は一切使わない。
2. **デザイントークンの外の値を一切書かない**
   HEX, px, フォントサイズを直書きしない。`G-DX_Design_System.md`（本ディレクトリに同梱）の
   トークンを CSS 変数／Tailwind クラスで参照する。
3. **要件定義（各システムの md / docs ）を単一ソースとする**
   画面・API・DB・権限は要件定義を正とし、差異が出た場合は md を先に更新してから実装する。
4. **既存 MVP 思想（履歴保持・監査ログ必須・運用負荷優先）を壊さない**
5. **絵文字を UI に出さない**

## 2. 参照する共通ドキュメント

本ディレクトリ（`ClaudeCodeルール/`）に置かれる共通ドキュメント：

- `CLAUDE.md`（本ファイル）— グラスト共通の最上位ルール
- `G-DX_Design_System.md` — デザイントークン／コンポーネント規定／ライティング／アクセシビリティの**正本**

各システムの CLAUDE.md からは、冒頭で必ず本ディレクトリを参照すること。
記述例：

```markdown
## 参照ドキュメント
- グラスト共通ルール: `../ClaudeCodeルール/CLAUDE.md`（最優先）
- デザインシステム: `../ClaudeCodeルール/G-DX_Design_System.md`（正本）
- 本システム要件: `./docs/...`
```

**正本の重複コピーを各リポジトリに置かない。** `G-DX_Design_System.md` は
本ディレクトリのものが唯一の正。各システムでカスタマイズが必要な場合は
本ディレクトリ側を更新し、変更履歴に追記する。

## 3. 技術スタック（標準）

新規システムは以下を**標準スタック**とする。逸脱する場合は事前にユーザー承認。

| レイヤー | 採用技術 |
|---|---|
| フレームワーク | Next.js 15（App Router） |
| 言語 | TypeScript（strict mode、`any` 原則禁止） |
| UI | React 19 |
| スタイリング | Tailwind CSS v4 |
| UI コンポーネント | shadcn/ui（G-DX トークンでカスタマイズ） |
| アイコン | lucide-react のみ（他アイコンライブラリの混在禁止） |
| DB | Supabase（PostgreSQL） |
| ホスティング | Vercel（または Cloudflare Pages） |
| フォーム | react-hook-form + zod |
| データ取得 | Server Components / Server Actions を第一選択、必要時のみ TanStack Query |
| 日時 | date-fns（dayjs / moment 禁止） |
| テスト | Vitest + Playwright |
| リンタ | ESLint（next/core-web-vitals）／ Prettier |

既存システム（g-dx_kanri など）が pnpm + turbo monorepo / 別構成を採用している場合は
その構成を尊重しつつ、UI レイヤーのデザイントークン適用は本ルールに合わせる。

**勝手に別ライブラリを入れない。** 追加が必要な場合は事前にユーザー承認。

## 4. デザイン原則（生成AI感を消すために）

### 4.1 やらないこと（Never）

- ❌ 紫→ピンク／青→紫などのグラデーション背景・ボタン
- ❌ ガラス風（backdrop-blur、半透明レイヤーの多用）
- ❌ ネオモーフィズム、極端な内側シャドウ
- ❌ `rounded-2xl` / `rounded-3xl` を業務画面の全要素に適用
- ❌ ページ全幅の巨大ヒーロー・巨大イラスト
- ❌ 絵文字を UI に配置
- ❌ 無駄な micro-interaction（hover で跳ねる／spring アニメ）
- ❌ ダークモード実装（明示的な要件がない限り対象外）
- ❌ 独自のアクセントカラー追加（デザイントークン外の色）
- ❌ カードに過剰な影（`shadow-xl` 以上）
- ❌ 一画面に 3 色以上のアクセントカラー
- ❌ 英語のキャッチコピー装飾（"Welcome!" 等）
- ❌ トーストの多用（原則：静かな確定感）

### 4.2 守ること（Always）

- ✅ `G-DX_Design_System.md §2` のトークンのみ使用
- ✅ 角丸は `RADIUS_S(4px)` / `RADIUS_M(6px)` / `RADIUS_L(8px)` / `RADIUS_FULL` のみ
- ✅ ボタンは Primary / Secondary / Danger / Text の 4 種のみ
- ✅ 1 画面の Primary ボタンは原則 1 つ
- ✅ テーブル中心のレイアウト（業務アプリのデータ密度）
- ✅ 見出しは `FONT_XXL / XL / L` の階層を守る
- ✅ ステータスは `StatusLabel` の決められた配色
- ✅ 日本語は「です・ます」調、ボタンは動詞原形（「保存する」「削除する」）
- ✅ フォーカスリング `SHADOW_OUTLINE_FOCUS` を全インタラクティブ要素に表示
- ✅ WCAG 2.1 AA 準拠（コントラスト比・キーボード操作）
- ✅ アイコンは意味が明確なときだけ。装飾目的の飾りアイコンは置かない

### 4.3 色の使い方（ブランド優先）

- Primary アクションは `MAIN`（#1a73e8 = GRUST_BLUE）一択
- Header 背景は `GRUST_NAVY`（#0a1628）
- アクセント（バッジ・チャートハイライト）に `GRUST_ACCENT`（#00c4cc）を少量
- それ以外の画面は 白 / `BACKGROUND` / `BORDER` / `TEXT_BLACK` / `TEXT_GREY` で構成
- チャート多色は `CHART_1〜10` の順で使用

### 4.4 レイアウト

- AppHeader（高さ 56px, `GRUST_NAVY`）＋ SideNav（240px, white）＋ コンテンツ
- コンテンツ最大幅 `WIDTH_CONTENT_MAX`（1200px）
- Base コンポーネント（白背景 + `SHADOW_S`）でグルーピング
- セクション間は `SPACE_L`（24px）以上

### 4.5 ライティング

- 専門用語は初出時に 1 行補足
- エラー文は「何が起きたか」＋「どうすればよいか」をセットで書く
- プレースホルダーでラベルを代用しない
- 英数字と日本語の間に半角スペースは入れない（Tailwind の `tracking-*` で調整しない）
- ボタンラベルは動詞で始める（「保存する」「削除する」「追加する」）

## 5. コーディング規約（共通最低ライン）

### 5.1 TypeScript

- `strict: true`, `noUncheckedIndexedAccess: true`
- `any` / `as unknown as` は原則禁止（やむを得ない場合は理由コメント必須）
- Zod スキーマを単一ソースに `z.infer` で型導出
- DB の型は Supabase の型生成を利用

### 5.2 命名

- ファイル: kebab-case（`rating-engine.ts`）
- コンポーネント: PascalCase（`CompetencyScoreForm.tsx`）
- 変数・関数: camelCase
- 型・interface: PascalCase
- DB カラム: snake_case（`effective_from`）
- 日本語変数・関数名は禁止

### 5.3 コメント

- **デフォルトは書かない**
- 書くのは「なぜ」。「何を」は名前で表現
- 要件 md への参照は `// see: docs/<file>.md §<番号>` 形式

### 5.4 エラーハンドリング

- ユーザーに見えるエラーは日本語
- 例外で制御フローを作らない（Result / Either 風の戻り値を採用）
- 権限エラーと入力バリデーションエラーは区別して UI 表示

## 6. セキュリティ共通方針

- 機微情報（評価・給与・個人情報・案件売上など）は RLS ＋アプリ層で二重防衛
- 環境変数は `.env.local`（gitignore）に保存、本番は Vercel の環境変数に登録
- ログイン成功／失敗、機微情報の閲覧イベントの監査ログを記録
- 制度パラメータ・金額・権限の変更は `audit_logs` に before/after JSON で保存
- 本番 DB への破壊的操作（drop / truncate / reset）はユーザー承認なしに実行しない

## 7. Claude Code への進め方の指示

### 7.1 作業着手時にやること

1. 各システムの CLAUDE.md と要件 md を読む
2. 影響するテーブル・画面・権限を列挙する
3. デザイントークンに該当がない UI 要素は「新規トークンが要るか要件の誤解か」を問う
4. 実装 → テスト → 動作確認（UI は必ずブラウザ確認）の順で進める

### 7.2 md と実装が食い違ったとき

- **勝手に実装で解釈しない**
- まずユーザーに「md 修正 or 実装調整」を問う
- 決まったら md を先に書き換えてから実装を直す

### 7.3 勝手にやらないこと

- 新しいパッケージの追加
- Supabase のスキーマを migration 経由以外で変更
- 要件定義にない機能の追加（"これもあったら便利そう" で増やさない）
- デザイントークン外の色・角丸・影の追加
- 本番 DB への破壊的操作

### 7.4 PR / コミット

- コミットメッセージ：50 字以内の日本語サマリ、本文に「なぜ」を 1〜3 行
- PR 説明テンプレ：
  - 目的（要件のどの md の何番か）
  - 変更範囲
  - 影響するテーブル／画面／権限
  - 動作確認手順
  - スクリーンショット（UI 変更時）
