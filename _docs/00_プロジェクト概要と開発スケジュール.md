# イデアグループ様 基幹業務システム再構築プロジェクト

## プロジェクト概要

### 背景
株式会社イデアグループ様（土地家屋調査士法人 イデアグループ）にて現在稼働している
**MS Access ベース「業務管理システム Ver.2」**および周辺の各種 Web/ローカルフォーム群を、
弊社の **G-DX プラットフォーム（Next.js/TypeScript/PostgreSQL）** と **Lark（Base / 承認 / メッセンジャー / Anycross）**
を組み合わせて SaaS 型のクラウド業務基盤に再構築する。

### 対象システム（画像フォルダベースで確認した範囲）
| # | 現行モジュール | 現行実装 | 移行先 |
|---|---|---|---|
| 1 | 業務管理システム Ver.2（Main Switchboard） | MS Access | **G-DX**（案件/受託簿/工程） |
| 2 | 現場フォーム（多タブの現場カルテ） | MS Access サブフォーム | **G-DX**（現場詳細画面） |
| 3 | 報酬額計算書入力フォーム（見積・請求・計算書） | MS Access + Excel | **G-DX**（見積/請求/帳票） |
| 4 | 立ち会い申請フォーム（名古屋市/北名古屋市/県） | MS Access + Word差込 | **G-DX**（申請ワークベンチ + PDF生成） |
| 5 | 立ち会い申請帳票（所有者一覧・委任状・挨拶状ほか） | Word/Excel テンプレート | **G-DX** で PDF 生成 + Lark Drive 保管 |
| 6 | 出退勤届け出システム（勤怠ポータル） | 社内 Web（127.0.0.1 運用） | **Lark Base + 承認フロー**（G-DX と社員マスタ連携） |
| 7 | 経費精算 | 社内 Web + TXT 出力 | **Lark Base + 承認フロー**（会計 TXT は Anycross で生成） |

### 利用者の想定
- **土地家屋調査士・測量担当**（現場）：現場カルテ入力、立会申請、進捗更新
- **調査士補助・事務**：受託簿入力、見積・請求書発行、顧客対応
- **管理部門（経理・総務）**：勤怠・経費承認、会計 TXT 出力
- **経営層**：ダッシュボードで受託状況・売上・進捗を把握
- **依頼先（外部）**：メールで進捗報告 PDF を受領（将来：外部ポータルで閲覧）

### プロジェクトのゴール
1. Access 固有のローカル依存・共有排他・データ散在を解消し、**クラウドで同時利用可能**にする
2. 現場からスマホ／タブレットで**現場カルテ・立会申請・写真登録**を行えるようにする
3. **見積 → 受託 → 進捗 → 立会 → 成果 → 請求 → 入金** を 1 本の案件ID で一気通貫に管理
4. 勤怠・経費は **Lark** で即稼働（既存 Lark 利用なら 1～2 ヶ月で先行リリース可能）
5. 会計ソフトへの **TXT エクスポート**、現行 Word/Excel 帳票フォーマットの **完全再現**

## 技術スタック（提案）

### G-DX 側（弊社標準）
- **フロントエンド**: Next.js (App Router) / TypeScript / TailwindCSS / shadcn-ui
- **バックエンド**: Next.js Route Handlers or NestJS / Prisma ORM
- **DB**: PostgreSQL（本番は Supabase or Aurora PostgreSQL）
- **認証**: Lark OAuth SSO（社員アカウント統一）+ ロールベース認可
- **ファイル**: S3 互換ストレージ（R2 or Supabase Storage）
- **PDF 生成**: Puppeteer / Playwright（HTML → PDF）もしくは `@react-pdf/renderer`
- **Word 差込**: `docxtemplater` で Word テンプレートに差込
- **住所・地番**: 不動産登記情報インポート（CSV / API）

### Lark 側
- **Lark Base**: 勤怠届出・経費精算の一次入力と参照
- **Lark 承認（Approval）**: 休暇／経費の多段承認フロー
- **Lark Anycross / Webhook**: G-DX ↔ Lark 双方向連携（社員マスタ同期、通知、ファイル連携）
- **Lark Drive**: 帳票 PDF の保管（G-DX からアップロード）
- **Lark Docs / Sheets**: 集計表や会議資料の共同編集（任意）

### データ移行
- 既存 Access `.mdb/.accdb` から CSV に一括エクスポート → 変換スクリプト（TypeScript）で PostgreSQL に投入
- 主要マスタ（社員・単価・法務局・依頼先）は先行移行。現場／土地／事件のトランザクションは並行稼働期間で段階移行

## リスクと前提

### リスク
| # | リスク | 影響 | 対策 |
|---|---|---|---|
| R1 | 既存 Access のロジック（VBA マクロ・クエリ）がブラックボックス | 工数増・業務事故 | フェーズ0で全 VBA・クエリを棚卸し、仕様書化 |
| R2 | 報酬額計算の単価体系・調整ルールが属人的 | 計算不一致で請求ミス | 実データでの並行計算検証を 1 ヶ月以上実施 |
| R3 | Word 帳票の完全再現（差込レイアウト） | 客先印象・事務負荷 | docxtemplater ベース、現行 Word を原本として利用 |
| R4 | 会計ソフトの TXT フォーマット変更 | 経理業務停止 | 現行 TXT 仕様を凍結・テスト自動化 |
| R5 | 現場のモバイル利用環境（電波・PDF表示） | 採用率低下 | 下書き保存・オフライン入力・軽量 UI |
| R6 | Lark と G-DX の社員マスタずれ | 承認フロー破綻 | Lark を SOT（Source of Truth）とし G-DX が同期 |

### 前提
- イデアグループ様は Lark 有償プランを契約済（または契約する）
- 既存 Access の管理者がリバース調査に協力可能
- 現行 Word/Excel テンプレート（.docx/.xlsx の原本）が入手可能
- 本番稼働前に 2 週間の並行稼働期間を確保できる
