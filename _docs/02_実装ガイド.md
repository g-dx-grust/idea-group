バージョン: v0.1 / 最終更新: 2026-04-24

このドキュメントは、要件定義書（`01_要件定義書.md`）に基づき、
開発チームが **何を・どう作るか** を示す技術ガイドです。

## 1. 役割分担（G-DX と Lark）

| 機能 | G-DX | Lark |
|---|---|---|
| 認証（SSO） | クライアント | **プロバイダ（OAuth 発行）** |
| 社員マスタ | 読み取り＋独自プロファイル拡張 | **SOT**（Source of Truth） |
| 現場／土地／登記事件 | **主** | - |
| 報酬額計算書／見積／請求／領収 | **主** | - |
| 立会申請／帳票 | **主** | Drive に PDF 保管（任意） |
| 勤怠届出 | 集計参照のみ | **主**（Base + 承認） |
| 経費精算 | 集計参照のみ | **主**（Base + 承認 + Anycross で TXT 生成） |
| 通知（チャット） | 起点 | **受信**（メッセージ送信経由） |
| ダッシュボード | **主** | 参考 Base ビュー |

原則: **台帳データは G-DX、人事ワークフローは Lark**。

## 5. フロントエンド実装方針

- **App Router** + **Server Components** を基本、帳票プレビュー等は Client Component
- **フォーム**: `react-hook-form` + `zod`
- **テーブル**: `@tanstack/react-table`
- **多タブ現場カルテ**: ルートレイアウトにタブナビ、子ルートで各タブを遅延ロード
- **状態管理**: サーバー状態は `TanStack Query`、UI ローカル状態は `zustand`
- **モバイル対応**: Tailwind のレスポンシブ、フォームを縦並びに、カメラ連携で写真即アップロード
- **印刷**: `window.print()` 用のプレビュー画面を別途用意、帳票は PDF を新タブ表示

## 6. 帳票生成パイプライン

### 6.1 PDF（A4 レイアウト固定）— Puppeteer
1. HTML/React テンプレートを Next.js のサーバーでレンダリング
2. BullMQ ワーカーが Puppeteer でレンダリング結果を PDF 化
3. S3 に保存、`documents` テーブルに記録、URL を返す
4. 進捗報告書はメール送信（Resend or SMTP）に PDF を添付

### 6.2 Word 差込（既存フォーマット再利用）— docxtemplater
- 現行の `.docx` テンプレートに `{{ }}` プレースホルダを挿入
- サーバーで docxtemplater にデータ差込 → `.docx` を返す
- ユーザーがそのまま編集可能（Word に慣れた事務向け）
- 必要に応じて LibreOffice でヘッドレス PDF 変換

### 6.3 帳票テンプレート配置
```
packages/pdf-templates/
├─ estimates/ 御見積書.docx
├─ invoices/ 御請求書.docx
├─ rewards/   報酬額計算書.xlsx
├─ site-visits/
│   ├─ 共通_隣接地所有者等一覧表.docx
```

## 7. Lark 側の実装手順

### 7.1 勤怠 Base（攻略）
1. Lark Base「勤怠届出」ベースを作成
   - フィールド: 従業員（Lark ユーザー）、届出日時、届出種別、開始日／区分、終了日／区分、日数、理由、業務内容、区分（予定／実績）、時間、ステータス
   - ビュー: 「社員ビュー（自分の届出）」「承認者ビュー（部下の PENDING）」「経理ビュー（経理承認待ち）」
2. 承認フローを「Lark 承認」で 2 段（上長 → 経理）で作成、Base と連動
3. Anycross で Base の INSERT/UPDATE を Webhook で G-DX に通知
4. G-DX 側 `AttendanceRequestMirror` を更新

### 7.2 経費 Base
1. Base「経費精算」（ヘッダ）＋ 子 Base「経費明細」
2. 入力フォームは Lark フォームで公開 → 社員は URL 1 つで使える
3. 承認フロー（上長 → 経理）
4. 経理承認後、Anycross スクリプトで会計 TXT を生成
   - 勘定科目マスタ（G-DX 側 DB）と Base の「購入物」をマッピング
   - TXT フォーマットは既存会計ソフト仕様（後で凍結）
5. 生成した TXT は Lark Drive と G-DX に保存

### 7.3 Lark ↔ G-DX 同期ポイント
| データ | 同期方向 | 頻度 |
|---|---|---|
| 社員マスタ | Lark → G-DX | 日次バッチ + 入退社時 Webhook |
| 組織 | Lark → G-DX | 日次 |
| 勤怠届出 | Lark → G-DX（ミラー） | 変更時 Webhook |
| 経費精算 | Lark → G-DX（ミラー） | 承認時 Webhook |
| 現場担当変更通知 | G-DX → Lark（チャット送信） | イベント時 |
| 請求書発行通知 | G-DX → Lark（チャット送信） | 発行時 |

## 8. Access → PostgreSQL データ移行手順

### 8.1 事前準備
1. `.accdb` のコピーを取得（運用中インスタンスに直接アクセスしない）
2. Windows 機 + ODBC ドライバで `mdb-export` or PowerShell で全テーブルを CSV 化
3. テーブル・フィールド一覧を差分スプレッドシートに整理（日本語名 → 英語論理名へのマッピング）

### 8.2 変換スクリプト
`scripts/migrate-access/` に TypeScript で変換スクリプトを実装。

```
scripts/migrate-access/
├─ 01_masters.ts       # 社員 → employees, 単価 → unit_prices ...
├─ 02_cases.ts         # 現場（T_現場） → cases
├─ 03_lands.ts         # 土地 → lands, 所有者 → land_owners
├─ 04_site_visits.ts   # 立会申請
├─ 05_rewards.ts       # 報酬額計算書＋明細
├─ 06_work_times.ts
├─ 07_attachments.ts   # 添付ファイルは S3 にアップロードしパスを保存
└─ run.ts              # 全体オーケストレーション（トランザクション）
```

### 8.3 検証
- マスタ件数一致確認
- 現場件数 × 業務区分の分布比較
- 売上合計（過去 12 ヶ月）が ±1 円で一致
- サンプル 50 件を目視確認

### 8.4 本番カットオーバー
1. フェーズ 4 初週: フルダンプ → 移行リハーサル → 差異ゼロを確認
2. 前週末: 現行 Access を読み取り専用にし、最終差分のみ Delta 移行
3. 月曜朝: G-DX 本番公開、Access は凍結バックアップとして保全

## 10. テスト戦略

3. **Week 4-8**: Lark 側の勤怠・経費を先行構築してデモ（効果が見えやすい）
4. **Week 8-20**: 現場管理（最大のコア）をスプリント化（2 週間スプリント × 6）
5. **Week 20-32**: 帳票・立会申請を並行スプリント（フロント寄りとバックエンド寄りで分担）
6. **Week 32-40**: データ移行リハーサル × 2 回、UAT
7. **Week 40-48**: 本稼働、ハイパーケア、改善
