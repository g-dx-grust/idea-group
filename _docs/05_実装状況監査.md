## 初回監査時点の結論

初回監査時点のディレクトリは、実行可能なシステム本体ではありません。

`_docs` 配下に要件定義・画面仕様・帳票仕様・Drizzle スキーマ案・一部 Next.js コンポーネント例はありますが、Next.js アプリとして起動するための `package.json`、`apps/web`、`tsconfig.json`、DB 接続、環境変数、マイグレーション、テスト、CI が存在しません。そのため、`pnpm install`、`pnpm dev`、型チェック、ブラウザ動作確認はいずれも実施できない状態です。

## コード上の注意点

| 箇所 | 内容 |
|---|---|
| `_docs/src/app/api/documents/generate/route.ts` | `docType` は `receipt` を受け付けるが、実際の `switch` では領収書を生成しない |
| `_docs/src/app/api/documents/generate/route.ts` | PDF を S3 にアップロードした後の DB 記録が見積・請求のみ。報酬額計算書と領収書は履歴保存されない |
| `_docs/src/app/api/documents/generate/route.ts` | 認証・認可・監査ログがないため、帳票生成 API としては業務利用不可 |
| `_docs/src/app/api/documents/generate/route.ts` | `@/lib/pdf/*`、`@/lib/storage` が存在しない |
| `_docs/src/components/rewards/RewardForm.tsx` | `rounded-lg`、`text-gray-*`、`text-blue-*`、`text-red-*` など、共通ルールのデザイントークン外 Tailwind が多い |
| `_docs/src/components/rewards/RewardForm.tsx` | 金額は `onBlur` 時のみ再計算。単価マスタ連動と保存時のサーバー側再計算がない |
| `_docs/src/components/rewards/RewardForm.tsx` | 計算結果の `subtotal`、`totalReward`、`taxAmount`、`total` を submit 値として確定する処理がない |
| `_docs/schema/idea-lands.ts` | `chibанIdx` に見た目が `a` のキリル文字が混入している。識別しづらく修正推奨 |
| `_docs/src/lib/wareki.ts` | `toWareciShort` は `Wareki` の typo と見られる |
| `_docs/02_実装ガイド.md` | DB 設計は Prisma と書かれているが、同梱スキーマは Drizzle。方針統一が必要 |

## 起動可能にするための最短手順

1. G-DX 標準構成で Next.js アプリ本体を作成する
2. `packages/database` を作成し、Drizzle か Prisma のどちらかに統一する
3. `_docs/schema` を実スキーマへ移し、マイグレーションを生成する
4. 認証・ユーザー・ロール・監査ログを先に実装する
5. 現場一覧、現場新規作成、現場詳細の最小 CRUD を実装する
6. 報酬額計算はサーバー側計算ロジックとテストを先に固める
7. 立会申請と帳票はテンプレート原本入手後に実装する
8. Lark 勤怠・経費は Lark 管理者権限と Base 設計確定後に構築する
